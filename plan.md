# Full-duplex акустический мессенджер (CSS + OFDM/QAM), Rust + Tauri 2.x

## Контекст

Задача хакатона (`task.md`): передавать цифровые данные между устройствами через аудиоканал (запись/воспроизведение звука), когда обычные средства связи недоступны. Оценка: эффективность передачи, устойчивость к помехам, качество реализации.

Репозиторий сейчас — чистый, немодифицированный скаффолд `create-tauri-app` (vanilla TS + Vite, Tauri v2, `src-tauri` — не workspace, только demo-команда `greet`). Уже сгенерирован Android-таргет (`src-tauri/gen/android`, package `com.tuhlopuz.sonic`), CI собирает и smoke-тестирует десктоп (Win/Linux/macOS) и Android-эмулятор. DSP/аудио-кода нет вообще — чистый лист.

Договорённости с пользователем (не пересматриваем):
1. **Duplex:** частотное разделение (FDD) на первую версию — делим полосу на два непересекающихся под-диапазона (A→B, B→A), без эхоподавления. Архитектура должна иметь чистый шов для последующего добавления общей полосы + адаптивного AEC (NLMS), без переписывания модуляторов.
2. **Платформы:** десктоп (Win/Linux/macOS) и Android — сразу, параллельно.
3. **Диапазоны/профили:** «слышимый» профиль (по умолчанию, штатные mic/speaker) и «ультразвуковой» профиль (~21–24 кГц, для отдельного трека — модулей на ESP32 с более качественными преобразователями). Оба профиля используют один и тот же протокол/кадр и оба режима модуляции — различаются только параметры полосы.
4. **Два режима модуляции:** CSS (Chirp Spread Spectrum, «надёжный») и OFDM+QAM («быстрый»), с авто-fallback с OFDM на CSS при деградации канала.

Цель плана — довести до работающей демонстрации: десктоп+Android мессенджер поверх штатного железа, с обоими режимами модуляции и объективными метриками (BER/throughput), и явно выделить ESP32-ультразвуковой трек и AEC как отдельные более поздние фазы.

## 2. Новые крейты

- **`crates/sonic-protocol`** — чистый Rust, без ОС/аудио зависимостей (без cpal/tauri). DSP-ядро, юнит-тестируемое без железа, портируемое в сторону будущей embedded-цели.
- **`crates/sonic-audio`** — cpal duplex-движок, ring-buffer мосты, (пока no-op) шов для эхоподавления. Зависит от `sonic-protocol` (трейты `Modem`/`DuplexScheme`).
- **`src-tauri`** — остаётся Tauri-приложением, добавляет path-зависимости на два крейта выше, хостит команды/события/состояние.

Один крейт `sonic-protocol` (не дробим на sonic-dsp/sonic-fec) — для хакатона достаточно модульных границ внутри, лишний workspace-оверхед не нужен.

### Ключевые архитектурные решения

**`DuplexScheme` — шов FDD→AEC.** Модуляторы (CSS/OFDM) никогда не знают про FDD vs shared-band — они только спрашивают `tx_band()/rx_band()/sample_rate()` у активной схемы и сдвигают baseband-сигнал в нужную полосу.
```rust
pub struct SubBand { pub center_hz: f32, pub bandwidth_hz: f32 }
pub trait DuplexScheme {
    fn tx_band(&self) -> SubBand;
    fn rx_band(&self) -> SubBand;
    fn sample_rate(&self) -> u32;
    fn echo_canceller(&self) -> Box<dyn EchoCanceller>; // сейчас NoopEchoCanceller
}
pub struct Fdd { pub role: Role, pub profile: Profile }
// позже: SharedBandAec, тот же трейт
```
`EchoCanceller` — no-op сейчас, но `pipeline.rs` (см. §3) с самого начала прокидывает точную копию воспроизводимого сигнала как reference-вход в canceller (даже если тот её отбрасывает) — иначе позже придётся переделывать пайплайн, а не только подставить NLMS-реализацию.

**Роль (A/B) — явный выбор пользователя.** Раз штатных каналов связи нет, нет и способа авто-согласовать, кто A, а кто B — это выбор в UI при старте сессии («Начать как A» / «Подключиться как B»).

**Заголовок кадра — всегда в робастной, mode-agnostic кодировке**, независимо от режима payload'а:
```
[Робастная преамбула][Робастный заголовок: ver|mode|len|seq|flags|CRC16][FEC-кодированный payload][CRC32]
```
Это снимает необходимость знать текущий режим заранее — приёмник декодирует заголовок робастно, потом узнаёт, каким демодулятором читать payload. На этом основан однозначный auto-fallback.

**FEC: Reed-Solomon (не свёрточный/Viterbi).** Ошибки демодуляции в акустике пакетные на уровне символа (плохой FFT-пик в CSS, повреждённая поднесущая в OFDM от всплеска шума) — RS (GF(256)) естественно исправляет это как один символ, а не как россыпь битовых ошибок (для чего лучше подходил бы Viterbi, но писать его с нуля — дорого по времени). Плюс блочный интерливер, чтобы широкий всплеск не убил больше `t` ошибок в одном кодовом слове.

Важный нюанс с крейтами: `reed-solomon-erasure` решает **другую** задачу (erasure — известные позиции потерь, как RAID), не годится напрямую для битовых ошибок в неизвестных позициях. Нужен настоящий error-correcting декодер: `reed-solomon` (mersinvald) или `reed_solomon_rs` (SohamJog, Berlekamp-Welch) — оба менее «мейнстримные», чем `reed-solomon-erasure`, стоит прочитать код перед тем как полагаться. План Б, если оба разочаруют: CRC на каждый шард + `reed-solomon-erasure` как внешний код (CRC-фейл = erasure).

**ARQ.** FDD даёт настоящий одновременный full-duplex — ACK идёт по обратной полосе не блокируя приём данных. `WindowedArq<const N: usize>` с самого начала: stop-and-wait v1 — это `WindowedArq<1>`, апгрейд до скользящего окна (N=4-8, повышает эффективность) не требует переписывания.

**Auto-fallback.** Считаем подряд идущие неудачи ARQ; после порога — переключаем исходящий `PhyMode` OFDM→CSS (с гистерезисом: N успешных CSS round-trip'ов перед пробным возвратом на OFDM, чтобы не осциллировать). Это сильный, наглядный демо-момент для критерия «устойчивость».

**Диапазоны (ориентировочно, финализировать эмпирически на Фазе 0):**
- Слышимый: guard 1.5–2.0 кГц | A→B 2.0–6.5 кГц | guard | B→A 7.0–11.5 кГц | guard.
- Ультразвуковой (обязателен sample rate 48кГц, Nyquist=24кГц): A→B 21.0–22.4 кГц | guard | B→A 22.6–24.0 кГц. Верхняя граница у самого Nyquist — АЧХ/анти-алиасинг ЦАП/АЦП может подъедать полосу, проверять эмпирически на реальном ESP32-железе, не считать данностью.

### Симулированный канал (`sim/`)
`awgn.rs` (гауссов шум на заданный SNR + самопроверка измеренного SNR), `multipath.rs` (FIR-свёртка — синтетическая экспоненциальная модель или измеренный RIR из WAV через `hound`), `clock_drift.rs` (дробный ресемплинг, эмулирующий рассинхрон частоты дискретизации двух независимых звуковых карт). Двойное назначение: это и `cargo test`-инфраструктура, и источник объективных цифр BER/PER/throughput-vs-SNR — вынести отдельно как `crates/sonic-protocol/examples/ber_sweep.rs` (свип SNR 0-30дБ × {CSS, OFDM} × пара настроек CP, вывод в CSV/таблицу) — это и есть предъявляемое жюри доказательство «эффективности/устойчивости».

**Важная акустическая специфика для OFDM:** звук идёт со скоростью ~343 м/с, поэтому отражения от стен даже в паре метров дают многомиллисекундный разброс задержек — это может быть намного больше, чем типичный RF-мультипас относительно длительности символа. Длина CP и длительность OFDM-символа должны браться из эмпирического замера импульsной характеристики комнаты (Фаза 0), а не угадываться — недооценённый CP проявляется как пол BER, который никакой FEC не лечит.

## 3. Аудио-IO (`crates/sonic-audio`)

```
device.rs        — cpal Host/Device enumeration, default_io_config(profile) — форсирует 48кГц для Ultrasonic
duplex.rs         — DuplexEngine: один cpal input + один cpal output stream, работают одновременно
pipeline.rs       — capture ring buf → EchoCanceller(no-op) → Modem::decode → mpsc наверх;
                     app → Modem::encode → playback ring buf, с отводом копии как AEC reference
platform/android.rs — RECORD_AUDIO runtime permission, проверка JNI/Activity контекста
platform/desktop.rs
```

Full-duplex приёма+воспроизведения на уровне железа — константа независимо от режима (FDD vs будущий shared-band+AEC решает только DSP-слой, какую полосу использовать и включён ли canceller).

**Ring buffers:** `rtrb` (SPSC, lock-free, без аллокаций в аудио-callback) — сырые сэмплы между cpal callback и потоком обработки; `crossbeam-channel` — для декодированных/исходящих `Frame` наверх, к Tauri-командам/событиям (там SPSC не критичен).

**Десктоп vs Android — ключевой риск.** Десктопные бэкенды cpal (WASAPI/CoreAudio/ALSA) — зрелый путь. Android — менее зрелый бэкенд cpal (AAudio via `ndk-context`) **плюс** Android требует runtime-разрешение `RECORD_AUDIO`, которое голый Rust получить не может — нужен либо мини Tauri-плагин на Kotlin, либо переиспользование паттерна из существующих community-плагинов (это реальная, задокументированная боль — `tauri-apps/tauri#10846`, не гипотетический риск).

**Спайк перед любой Android DSP-работой:** отдельная ветка, cpal + мини permission-плагин, запрос `RECORD_AUDIO` в рантайме, открытие input stream на реальном устройстве, логирование RMS захваченных сэмплов в logcat. **CI-эмулятор загружается с `-noaudio`** — это НЕ автоматизируется в CI, только ручная проверка на реальном устройстве.

## 5. Фазы

- **Фаза 0 — спайки и скаффолдинг.** Конверсия в workspace (§1) + регрессионная проверка (`cargo build`, `npm run tauri dev`, `npm run tauri android build`); cpal-в-Tauri-Android спайк; замер АЧХ реального железа (лэптоп+телефон) для финализации слышимого диапазона и раннего ответа — тянет ли штатное железо хоть что-то на 21-24кГц; замер импульсной характеристики комнаты для расчёта CP OFDM. Демо ещё нет, но всё дальнейшее зависит от этих цифр.
- **Фаза 1 — корректность DSP-ядра**, без железа, чистый `cargo test`: bandplan, framing, FEC, ARQ, sim/, затем полный CSS-модем. Демо: два процесса обмениваются кадрами через симулированный канал или WAV-петлю (`hound`).
- **Фаза 2 — реальный звук, десктоп-петля.** `sonic-audio` duplex-движок + pipeline с шовом под AEC (проверяет, что абстракция компилируется end-to-end, даже когда FDD её не использует). Демо: два физически разнесённых устройства обмениваются текстом по воздуху, режим CSS.
- **Фаза 3 — интеграция с Tauri + чат UI.** Команды/события/состояние, минимальный фронтенд. Демо: полный чат, CSS, только десктоп.
- **Фаза 4 — Android.** Аудио-путь на находках Фазы 0, разрешение в манифесте, тест на реальном устройстве (не автоматизируется в CI). Демо: телефон↔лэптоп, слышимый CSS.
- **Фаза 5 — OFDM+QAM + auto-fallback.** QPSK сначала, CP/pilot из замеров Фазы 0, переключение режима, `AutoFallback`. Демо: заметно выше throughput в тишине, заметный автоматический откат на CSS при наведённой помехе — сильный момент для критерия «устойчивость».
- **Фаза 6 — полировка.** 16-QAM, апгрейд ARQ до скользящего окна, UI телеметрии, отчёт `ber_sweep` как предъявляемое доказательство эффективности/устойчивости.
- **Фаза 7 (явно позже/stretch, отдельная поставка) — ультразвуковой профиль + ESP32.** Подтвердить (по Фазе 0), реально ли штатное железо тянет 21-24кГц; дальше — отдельная embedded-кодовая база (esp-hal/esp-idf-hal либо C/Arduino), переиспользующая *дизайн* кадра/модуляции как спецификацию, не крейт напрямую.
- **Фаза 8 (явно позже/stretch) — shared-band AEC.** NLMS `EchoCanceller` + `SharedBandAec: DuplexScheme` поверх готового шва — по дизайну не требует правок в модуляторах.

## 6. Тестирование/CI

`cargo test -p sonic-protocol` (без звукового железа):
- `framing.rs` — round-trip, CRC ловит порчу, некорректный вход отклоняется корректно.
- `fec/` — RS round-trip без ошибок; до `t` символьных ошибок → верное декодирование; `t+1` → **явно детектируемый отказ**, не молчаливое неверное декодирование (критично — от этого зависит ARQ); интерливер — identity round-trip, разбивает синтетический burst на ≤1 ошибку на кодовое слово.
- `modem/css.rs` — 0-шум = 100% точность символов; BER-свип по AWGN с порогом; устойчивость к CFO и рассинхрону частоты дискретизации (`clock_drift.rs`).
- `modem/ofdm.rs` — детекция преамбулы при сдвиге таймингов; корректность эквалайзера на синтетическом мультипасе; BER-свип по паре длин CP — явно показывает пол BER при CP < разброса задержек (сам тест демонстрирует, зачем нужен замер RIR из Фазы 0).
- `arq.rs` — симулированные потери/порча вызывают ретрансмиссию; `AutoFallback` срабатывает по порогу и восстанавливается.

Тесты `sonic-audio`, открывающие реальные потоки, — под `#[ignore]` или feature-gate (`real-audio-tests`), чтобы `cargo test --workspace` оставался безопасным на headless CI-раннерах без звукового устройства.

CI-правки: см. §1 (новый job `unit-tests`, правки `workspaces:` строк в `ci.yml`/`release.yml`). Существующие smoke-тесты и Android build не трогаются.

## 7. Ключевые риски

1. **cpal + Android внутри Tauri** — незрелый бэкенд + Java-side runtime-разрешение, которое голый Rust не получит сам (задокументированная проблема, tauri-apps/tauri#10846). Митигируется спайком на Фазе 0 и ручной (не CI) проверкой на реальном устройстве.
2. **Импульсная характеристика комнаты / размер CP для OFDM** — акустический разброс задержек может быть намного больше RF-аналогов относительно скорости символов; нужен реальный замер, не догадка.
3. **Тянет ли штатное железо 21-24кГц вообще** — штатные динамики/микрофоны лэптопов/телефонов часто заваливаются задолго до 21кГц; спайк Фазы 0 должен дать ответ достаточно рано, чтобы перенаправить усилия, если ответ «нет».
4. **Зрелость крейтов Reed-Solomon** — крейты с настоящей коррекцией ошибок (не erasure-only) менее активно поддерживаются; заложить время на вычитывание исходников перед тем, как полагаться, и держать CRC-per-shard + `reed-solomon-erasure` как задокументированный План Б.
5. **Рассинхрон частоты дискретизации** двух независимых звуковых карт — заложить `clock_drift.rs` в стандартный свип тестов заранее, не как запоздалую доработку; устойчивость к этому — одно из главных преимуществ CSS перед OFDM, подчеркнуть в демо.
6. **Приватность микрофона Windows** может молча блокировать захват cpal без видимой ошибки — проверить на основной машине разработки заранее, дёшево поймать, легко потерять день на отладку иначе.

**Крейты:** высокая уверенность — `cpal`, `rustfft`, `rtrb`, `crc32fast`, `crossbeam-channel`, `hound`. Требуют проверки перед использованием — `reed-solomon`/`reed_solomon_rs` (менее активно поддерживаются, но реальны и рабочие), `reed-solomon-erasure` (решает другую задачу — erasure, не error-correction), `rubato` (ресемплинг, опционально для Фазы 2 — проверить актуальность поддержки).
