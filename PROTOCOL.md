# ADL/1 — Acoustic Data Link Protocol
Спецификация протокола full-duplex передачи данных через аудиоканал.
Стек реализации: Rust + Tauri 2.x.

---

## 1. Общая архитектура

Протокол построен по классической многоуровневой схеме, адаптированной под звуковой канал:

```
┌─────────────────────────────────────────┐
│  APP      сообщения чата, файлы, чанки   │
├─────────────────────────────────────────┤
│  SESSION  Noise-handshake, шифрование,   │
│           ключи, реплей-защита           │
├─────────────────────────────────────────┤
│  MAC      адресация, ACK/NACK, ARQ,      │
│           sequence numbers, flow control │
├─────────────────────────────────────────┤
│  PHY      модуляция (CSS / OFDM+QAM),    │
│           FEC, синхронизация, framing    │
├─────────────────────────────────────────┤
│  AUDIO    cpal capture/playback,         │
│           ring-buffer, resampling        │
└─────────────────────────────────────────┘
```

Ключевая идея: **каждый PHY-кадр, независимо от режима данных, снабжается заголовком, всегда закодированным в CSS**. Это даёт надёжную точку синхронизации и определения длины/режима даже когда полезная нагрузка идёт в хрупком OFDM-режиме. OFDM никогда не должен "гадать" длину кадра — она всегда приходит по устойчивому каналу.

## 2. Физический канал и full-duplex

### 2.1 Проблема дуплекса

У вас нет отдельных проводов "туда" и "обратно" — один физический эфир. Full duplex через акустику получить можно двумя способами:

1. **AEC (акустическое эхоподавление)** — устройство знает, что оно само излучает, и вычитает это из принятого сигнала (адаптивный фильтр, как в VoIP). Сложно, чувствительно к нелинейности динамика/микрофона, требует оценки задержки акустического пути в реальном времени.
2. **FDD (частотное разделение)** — каждое направление передачи получает свою полосу частот, они не пересекаются, поэтому эхо собственного сигнала просто фильтруется полосовым фильтром на приёме.

**Решение: FDD.** Он на порядок проще, детерминирован и полностью укладывается в "честную" реализацию за разумное время. AEC можно вынести в roadmap как v2-фичу для расширения полосы.

### 2.2 Разбиение спектра

Рабочий диапазон ограничен реальными характеристиками дешёвых динамиков/микрофонов ноутбуков и телефонов: заметный спад АЧХ ниже ~150 Гц и выше ~16–17 кГц, плюс дыхание/шум помещения концентрируется на низах.

Используем диапазон **500 Гц – 15000 Гц**, sample rate 48000 Гц (стандарт для cpal/большинства аудиокарт).

Роль в сессии (initiator/responder) определяется на этапе handshake (см. §7). Инициатор всегда получает "нижнюю" полосу для передачи, респондер — "верхнюю".

```
500Hz            7200Hz  7800Hz            15000Hz
│  Initiator→Resp  │ guard │  Responder→Init  │
│    (band A)       │  band │     (band B)     │
```

- Полоса A: 500–7200 Гц (~6700 Гц)
- Guard band: 7200–7800 Гц (600 Гц, защита от протечки спектра/фильтрации)
- Полоса B: 7800–15000 Гц (~7200 Гц)

Каждая полоса делится ещё раз внутри себя между CSS-заголовком (нижняя часть, самая узкая и надёжная) и телом кадра (CSS или OFDM в зависимости от режима).

### 2.3 Почему не просто TDD (разговор по очереди)

TDD (half-duplex, "уоки-токи") проще, но не даёт full duplex, который явно требуется в задаче, и хуже с ACK/NACK — подтверждения всегда идут с задержкой на смену направления. FDD же позволяет слать ACK встречным потоком, пока идут данные — это критично для скорости в OFDM-режиме.

## 3. Синхронизация

Общая для обоих режимов проблема: приёмник не знает, когда начался кадр, и часы TX/RX (sample clock) слегка расходятся (обычно десятки–сотни ppm на дешёвых звуковых картах).

Синхронизация в три этапа:

1. **Грубая (энергетическая)** — скользящее окно RMS-энергии в рабочей полосе, детектор фронта.
2. **Точная (корреляционная)** — corss-корреляция входного сигнала с известной преамбулой:
   - для CSS: базовый up-chirp (см. §4),
   - для OFDM: Zadoff-Chu последовательность (постоянная амплитуда, острый пик автокорреляции, устойчива к частотному сдвигу).
   Пик корреляции даёт точный сэмпл начала кадра.
3. **Тонкая (фазовая/частотная)** — по двум одинаковым OFDM-пилот-символам в преамбуле оценивается остаточный частотный сдвиг (CFO) и корректируется поворотом фазы по поднесущим; по чирпам в CSS — оценивается временной дрейф символьной решётки.

Дрейф sample clock компенсируется периодической пересинхронизацией: раз в N символов проверяется положение пика корреляции по известным пилотам/чирпам и корректируется индекс окна (аналог "timing recovery" в модемах).

## 4. Режим CSS (устойчивый, низкая скорость)

Идея — как в LoRa: информация кодируется не амплитудой/фазой, а **циклическим сдвигом линейного чирпа**. Это даёт большой processing gain: энергия символа размазана по всей полосе и по времени, что делает канал устойчивым к узкополосным помехам, щелчкам, голосу в комнате и т.д.

### 4.1 Параметры

| Параметр | Значение | Комментарий |
|---|---|---|
| Полоса символа (BW) | 1600 Гц | нижняя часть band A/B |
| Spreading Factor (SF) | 8 (адаптивно 6–10) | бит/символ = SF |
| Длительность символа | 2^SF / BW | при SF=8 → 160 мс |
| Символьная скорость | BW / 2^SF | при SF=8 → ~6.25 symbol/s |
| Полезная скорость | SF × symbol_rate | при SF=8 → ~50 bit/s (до FEC) |

SF адаптивен: при хорошем SNR можно опускаться до SF=6 (~200 bit/s), при плохом — подниматься до SF=10 (~12 bit/s) для максимальной устойчивости. Решение принимает получатель на основе измеренного SNR преамбулы и передаёт желаемый SF в управляющем кадре.

### 4.2 Символ

Базовый up-chirp — сигнал, частота которого линейно растёт от f0 до f0+BW за время символа. Символ, кодирующий значение `s ∈ [0, 2^SF)`, — это тот же чирп, циклически сдвинутый на `s / 2^SF` от начала.

Демодуляция: перемножить принятый сигнал с "de-chirp" (обратным чирпом), взять FFT — позиция пика по частоте прямо даёт значение `s`. Это дёшево по CPU и устойчиво к шуму (интеграция по всей длительности символа = accumulated gain).

### 4.3 Преамбула CSS-кадра

```
[8 × up-chirp]  [2 × down-chirp]  [sync-symbol]  [header]  [payload...]
```

- 8 одинаковых up-chirp — грубая детекция + оценка частотного сдвига.
- 2 down-chirp — маркер конца преамбулы (пересечение up/down даёт острый пик — точная временная привязка).
- sync-symbol — фиксированное известное значение (магическое число 0x2B), защита от ложного срабатывания на шуме/музыке.

## 5. Режим OFDM+QAM (быстрый)

### 5.1 Параметры

| Параметр | Значение |
|---|---|
| Sample rate | 48000 Гц |
| Размер FFT (N) | 1024 |
| Разнос поднесущих | 46.875 Гц |
| Циклический префикс (CP) | 128 сэмплов (2.67 мс) — покрывает реверберацию типичной комнаты (RT60 до ~150–200 мс с запасом на затухание хвоста) |
| Длительность OFDM-символа | (1024+128)/48000 = 24 мс |
| Полоса под данные | ~6700 Гц (band A) → ~140 полезных поднесущих |
| Пилот-поднесущие | каждая 8-я (для оценки канала и фазы) |
| Модуляция поднесущих | адаптивно QPSK / 16-QAM / 64-QAM |

Расчёт скорости (band A, 16-QAM, ~120 data-поднесущих из 140, 20 — пилоты):
`120 × 4 бита / 0.024 с ≈ 20 кбит/с` сырых, до FEC и служебных полей. С учётом FEC (см. §6) и заголовков — ожидаем **8–12 кбит/с полезной нагрузки** в каждую сторону в хороших условиях, что уже прилично для акустики.

### 5.2 Адаптивная модуляция (bit-loading light)

Не полноценный per-subcarrier bit-loading (это overkill для MVP), а простой **адаптивный выбор одного созвездия на весь кадр**:

1. Приёмник постоянно оценивает SNR по пилотам последнего успешно принятого кадра.
2. Раз в K кадров отправляет управляющий CSS-кадр `MODE_REQUEST` с желаемым созвездием.
3. Отправитель переключается начиная со следующего кадра.

Ступени: QPSK (2 бита/поднесущая, самый устойчивый) → 16-QAM (4 бита) → 64-QAM (6 бит, только при высоком SNR и близком расстоянии).

### 5.3 Преамбула OFDM-кадра

```
[Zadoff-Chu sync symbol] [2× identical pilot OFDM symbol] [header(CSS)] [OFDM data symbols...]
```

Два одинаковых пилотных символа подряд — классический приём Schmidl-Cox для грубой временной/частотной синхронизации через автокорреляцию половинок символа.

## 6. Общий формат PHY-кадра

```
┌───────────┬──────────────────┬─────────────────────────┬──────────┐
│ Preamble  │  Header (CSS!)   │  Payload (CSS или OFDM)  │  Tail    │
└───────────┴──────────────────┴─────────────────────────┴──────────┘
```

### 6.1 Header (всегда CSS, SF=10 — максимальная надёжность), 6 байт до FEC:

| Поле | Биты | Описание |
|---|---|---|
| Magic | 8 | 0x2B — защита от ложных срабатываний |
| Version | 4 | версия протокола = 1 |
| Mode | 2 | 00=CSS, 01=OFDM-QPSK, 10=OFDM-16QAM, 11=OFDM-64QAM |
| Direction | 1 | 0=A→B, 1=B→A (защита от приёма собственного эха, если просочилось через guard band) |
| Reserved | 1 | — |
| PayloadLen | 12 | длина payload в байтах (до 4095) |
| HeaderCRC | 8 | CRC-8 самого заголовка |

Заголовок кодируется с (12,6) блочным кодом Голея или тройным повтором + мажоритарное голосование (проще в реализации, достаточно для 6 байт) — цена по времени мизерна (это единственная часть кадра, где лишние миллисекунды не жалко, т.к. от него зависит вся дальнейшая декодировка).

### 6.2 FEC полезной нагрузки

- **CSS-режим**: свёрточный код R=1/2, K=7 (стандартный, как в старых модемах/спутниковой связи) + Viterbi-декодер на приёме. Даёт ~5 дБ выигрыша ценой удвоения символов.
- **OFDM-режим**: Reed-Solomon (255,223) над байтами + битовый интерливер по поднесущим и по времени (разносит пакетные ошибки от глубоких замираний/щелчков по разным RS-словам). RS выбран, т.к. хорошо справляется с "пакетными" ошибками — а в акустике ошибки почти всегда пакетные (щелчок, скрип стула, дверь).

### 6.3 Tail

CRC-32 всего PHY-кадра (payload + header, после FEC-декодирования) — финальная проверка целостности перед передачей выше по стеку. Если CRC не сошёлся — кадр молча отбрасывается, инициируется NACK на MAC-уровне по таймауту.

## 7. MAC-уровень: адресация, ACK/NACK, ARQ

### 7.1 Идентификация устройств

- При первом запуске каждое устройство генерирует X25519-ключевую пару (статический ключ) через `snow`/`ring`.
- **DeviceID** = первые 4 байта BLAKE2s(pubkey) — используется как короткий адрес в каждом MAC-кадре.
- Полный публичный ключ передаётся один раз при handshake (§8) и используется для проверки — DeviceID это лишь адрес для маршрутизации в самом канале, не замена аутентификации.

### 7.2 Формат MAC-кадра (это и есть "payload" PHY-кадра до шифрования верхних полей)

```
┌────────┬──────┬─────────┬─────────┬─────────┬───────────────┬─────┐
│ Type(4)│Ver(4)│ SrcID(32)│ DstID(32)│ Seq(16) │ AckInfo(var)  │Data │
└────────┴──────┴─────────┴─────────┴─────────┴───────────────┴─────┘
```

**Type**: `HELLO`, `HELLO_ACK`, `HANDSHAKE`, `MODE_REQUEST`, `DATA`, `ACK`, `NACK`, `PING`, `BYE`.

**AckInfo** (присутствует для DATA/ACK/NACK):
- `CumulativeAck` (16 бит) — последний подряд идущий успешно принятый seq.
- `SackBitmap` (32 бита) — битовая маска следующих 32 seq после cumulative (селективное подтверждение, аналог TCP SACK) — позволяет одним кадром подтвердить/отвергнуть целое окно, не тратя отдельный ACK на каждый пакет.

### 7.3 ARQ (Automatic Repeat reQuest)

- **Sliding window**, selective repeat.
  - Размер окна для CSS-режима: 4 кадра (канал и так медленный, большой window не нужен и увеличивает буферизацию задержки).
  - Размер окна для OFDM-режима: 32 кадра.
- Отправитель шлёт кадры без ожидания ACK на каждый (пока не выйдет за границу окна), встречным потоком (в другой FDD-полосе) прилетают ACK/NACK.
- **NACK** высылается получателем сразу, если обнаружена дыра в последовательности (пришёл seq N+2, а N+1 не приходил дольше 2 символьных длительностей) — не дожидаясь таймаута отправителя. Это экономит время по сравнению с чистым timeout-based ARQ.
- **Таймаут** отправителя = 3 × RTT_estimate (RTT меряется на handshake и обновляется скользящим средним, как в TCP: `RTT = 0.875×RTT + 0.125×sample`).
- Экспоненциальный backoff при повторных потерях (защита от коллапса канала, если помеха длительная — например, кто-то включил музыку).
- После 8 неудачных повторов подряд — canal considered lost, MAC уходит в `LINK_DOWN`, приложение уведомляется.

### 7.4 Flow control

Простой credit-based: получатель в каждом ACK сообщает `WindowAvailable` — сколько кадров ещё готов принять (буфер decode-очереди ограничен) — защита от переполнения на медленном CSS-режиме, если верхний уровень пытается слать слишком быстро.

## 8. Session-уровень: шифрование и аутентификация

### 8.1 Handshake — Noise_XX

Выбран паттерн **Noise_XX** (через crate `snow`), потому что:
- обе стороны взаимно аутентифицируются статическими ключами,
- статические ключи передаются в рамках хендшейка (не нужен предварительный обмен ключами — удобно для "нашли друг друга по звуку и сразу начали"),
- даёт forward secrecy за счёт эфемерных ключей.

Крипто-примитивы: `X25519` (DH), `ChaCha20-Poly1305` (AEAD — выбран вместо AES-GCM, т.к. без аппаратного AES-NI ChaCha20 быстрее и одинаково безопасен, что важно, если Tauri-приложение крутится на слабом устройстве), `BLAKE2s` (hash).

Три сообщения Noise_XX (`e`, `e,ee,s,es`, `s,se`) передаются как обычные MAC-кадры типа `HANDSHAKE` через CSS-режим (максимальная надёжность на самом хрупком этапе соединения).

### 8.2 Защита от MITM — Short Authentication String (SAS)

Акустический канал по своей природе widely broadcast — ключи, полученные "из воздуха", теоретически подвержены активному MITM (третье устройство тоже слушает и говорит). Чистый Noise_XX без внешнего канала верификации даёт только TOFU (trust-on-first-use), не полную защиту.

Решение: после хендшейка обе стороны вычисляют `SAS = BLAKE2s(handshake_hash)[0:4 байта]` → отображают как **6-значное число на экране обоих устройств** (как Bluetooth/Signal). Пользователи глазами сверяют числа. Пока не подтверждено (кнопка "Совпадает" в Tauri UI) — сессия помечена `UNVERIFIED`, приложение показывает предупреждение, но может (по желанию пользователя) продолжать работу — это UX-компромисс, не блокирующий требование.

### 8.3 Кадры данных

Каждый `DATA` MAC-кадр после handshake шифруется:

```
ciphertext = ChaCha20Poly1305(key=session_key, nonce=counter, aad=SrcID||DstID||Seq, plaintext=app_data)
```

- `nonce` = монотонный счётчик (per-direction, 8 байт) — никогда не переиспользуется, что и требует ChaCha20-Poly1305.
- `AAD` включает адреса и seq — защита от replay/подмены адреса без расшифровки полезной нагрузки.
- Ре-кей: каждые 2^20 кадров или 30 минут — новый Noise-хендшейк (rekey) поверх уже установленного канала, без обрыва сессии (аналог TLS 1.3 KeyUpdate).

## 9. Адаптация режима (CSS ↔ OFDM)

- Сессия **всегда стартует в CSS** — discovery, handshake, SAS — весь "хрупкий" этап идёт по самому надёжному режиму.
- После `LINK_UP` отправитель по умолчанию пробует **пробный OFDM-кадр** (QPSK) с тестовыми данными.
- Если он подтверждён (ACK) — канал переходит в OFDM для объёмных данных (файлы, длинные сообщения); короткие сообщения (текст в чате) можно слать и в CSS без переключения — решение принимает приложение по размеру payload (порог, например, 64 байта).
- **Graceful degradation**: если 3 OFDM-кадра подряд не подтверждены — автоматический откат: сначала понижается созвездие (64→16→QPSK), если и QPSK не идёт — полный откат в CSS до восстановления канала (по CSS-пробникам раз в несколько секунд OFDM пробуется заново).
- Все переключения режима явно согласуются кадром `MODE_REQUEST`/`MODE_ACK` — получатель не переключает демодулятор "по догадке", а только по явной команде + подтверждению, чтобы не терять синхронизацию.

## 10. Конечный автомат сессии

```
IDLE
  │  start()
  ▼
DISCOVERY  ── periodic HELLO beacon (CSS) ──▶ (ждём HELLO_ACK от пира)
  │  HELLO_ACK получен
  ▼
HANDSHAKE  ── Noise_XX (3 msg, CSS) ──▶
  │  успешно, ключи установлены
  ▼
SAS_PENDING  ── ждём подтверждения пользователя (или auto-continue с warning)
  │
  ▼
LINK_UP  ◄──────────────┐
  │  data / ack / nack   │ mode switch success/fail
  ├─ CSS_ACTIVE ─────────┤
  ├─ OFDM_ACTIVE ────────┘
  │
  │  8 подряд неудачных ARQ / явный BYE
  ▼
LINK_DOWN → назад в DISCOVERY (авто-reconnect) или IDLE (по команде пользователя)
```

| Noise Protocol handshake | `snow` |
| AEAD-шифрование | `chacha20poly1305` |
| Хэши | `blake2` |
| Reed-Solomon FEC | `reed-solomon-erasure` или `reed-solomon-simd` |
| Свёрточный код / Viterbi (CSS) | написать самостоятельно (компактно, ~150 строк) или `viterbi_decoder` |
| CRC | `crc32fast`, `crc` |
| Асинхронность / очереди между потоками | `tokio` + `crossbeam-channel` |
| Ресемплинг (если разные девайсы дают разный sample rate) | `rubato` |

### 11.2 Архитектура потоков

```
[cpal input callback thread] --ring buffer--> [DSP/PHY thread] --mpsc--> [MAC/ARQ thread] --> [Session/crypto] --> Tauri event --> UI
[UI команда] --> Tauri command --> [Session] --> [MAC] --> [PHY encode] --ring buffer--> [cpal output callback thread]
```

Важно: **audio callback (cpal) должен быть lock-free и не аллоцировать** — вся тяжёлая обработка (FFT, Viterbi, крипто) уходит в отдельный поток через `crossbeam-channel` / `ringbuf` crate. Иначе будут дропы сэмплов и щелчки при GC/аллокациях в реальном времени.

### 11.3 Tauri-специфика

- Rust-бэкенд держит весь протокольный стек и модель состояний (state machine из §10) в `tauri::State`, обёрнутое в `Arc<Mutex<...>>` или через `tokio::sync::mpsc` команды.
- Фронтенд получает статус (уровень сигнала, SNR, текущий режим, RTT, статистика ретрансмиссий) через `emit` события — полезно сразу визуализировать спектрограмму принимаемого сигнала (хорошо смотрится на защите и реально помогает отлаживать).
- Команды из UI (`send_message`, `send_file`, `pair_device`, `confirm_sas`) — обычные `#[tauri::command]`.

### 11.4 Порядок разработки (рекомендуемый)

1. Audio loopback (запись/воспроизведение, ring buffer) без модуляции — просто убедиться, что I/O стабилен.
2. CSS-модем (мод/демод, преамбула, header) на loopback (виртуальный кабель / прямой пайп сэмплов) без реального акустического канала.
3. CSS через реальный воздух (динамик→микрофон), измерить реальный BER/SNR.
4. MAC + ARQ + FEC поверх CSS.
5. Noise-хендшейк + шифрование.
6. OFDM-модем (тоже сначала на loopback, потом по воздуху).
7. FDD-разделение и одновременный full-duplex тест.
8. Адаптивное переключение режимов + graceful degradation.
9. UI в Tauri, метрики, тестирование под помехами (музыка, шум, движение микрофона).

## 12. Метрики для оценки решения (под критерии жюри)

- **Эффективность**: измеренная полезная скорость (бит/с) отдельно для CSS и OFDM режимов, до и после FEC/ARQ overhead.
- **Устойчивость**: BER/PER (packet error rate) при контролируемых помехах — белый шум разного уровня, музыка, речь, хлопки, движение устройств друг относительно друга. Отдельно замерить: работает ли автоматический откат OFDM→CSS.
- **Качество реализации**: чистота протокольных слоёв (легко тестировать PHY отдельно от MAC отдельно от крипто — юнит-тесты на loopback-сэмплах без реального аудио), корректность ARQ (не теряет и не дублирует данные при принудительном дропе кадров в тесте), защищённость хендшейка (проверка SAS, защита от replay через nonce/AAD).
